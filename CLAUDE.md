# BoardScore — Project Guide

iOS board-game scoring app. iOS 26, SwiftUI, SwiftData, Liquid Glass, "Warm Table"
visual style. The Xcode project is generated by **xcodegen** from `app/project.yml`
— never hand-edit `GameScoring.xcodeproj`.

## Commands

Run from `app/`:

```bash
xcodegen generate                 # regenerate the project after adding/removing files
# build
xcodebuild -project GameScoring.xcodeproj -scheme GameScoring \
  -sdk iphonesimulator -destination "platform=iOS Simulator,name=iPhone 17 Pro,OS=26.4" \
  -configuration Debug build CODE_SIGNING_ALLOWED=NO
# test
xcodebuild test -project GameScoring.xcodeproj -scheme GameScoring \
  -sdk iphonesimulator -destination "platform=iOS Simulator,name=iPhone 17 Pro,OS=26.4" \
  CODE_SIGNING_ALLOWED=NO
```

After adding a new `.swift` file (including new test files), run `xcodegen generate`
before building or the file won't be in the target.

## Versioning — required

The app shows **two versions** on the Home tab footer and in the About sheet
(gear in the Shelf nav bar), both read from `Bundle.main` via `AppVersion`:

- **Release version** — `MARKETING_VERSION` in `app/project.yml` →
  `CFBundleShortVersionString` (e.g. `1.0.1`). Bump per release.
- **Dev build** — `CURRENT_PROJECT_VERSION` in `app/project.yml` →
  `CFBundleVersion` (an integer). **Bump this by 1 on every commit** that changes
  app code, in the same commit. Read both at runtime from `Bundle.main`; never
  hardcode the version in Swift.

So the commit workflow is: make changes → increment `CURRENT_PROJECT_VERSION` in
`app/project.yml` → `xcodegen generate` → build/test → commit. The footer reads
`"BoardScore \(release) · build \(dev)"`.

## Unit testing — required

Unit tests are a standing requirement, not an afterthought:

- **Every milestone/PR that adds or changes core logic must add or extend unit
  tests in the same change.** Core logic = models, scoring (`ScoringGame`
  conformers), services (`SessionService`), and any value transformation.
- **Tests must pass before committing.** Run the `xcodebuild test` command above
  and confirm zero failures. Do not commit a red suite.
- Tests live in `app/GameScoringTests/`, use **Swift Testing** (`import Testing`,
  `@Test`, `#expect`) — not XCTest — and `@testable import GameScoring`.
- Cover behavior and edge cases, not just the happy path (e.g. ties, empty
  inputs, wildcard distribution, the single-in-progress-session rule).
- UI views don't need unit tests, but the logic they call does — keep that logic
  in models/services so it's testable.

### UI tests (XCUITest)

- Flows live in `app/GameScoringUITests/` (target `GameScoringUITests`). Add a UI
  test when a user-facing flow changes (scoring, results, resume, etc.).
- The app honours launch arguments for deterministic state (see
  `GameScoringApp.makeContainer`): `-uitestClean` (fresh temp store),
  `-uitestSeed` (roster Ada/Boris/Chen/Dee), `-uitestSeedInProgress` (also an
  in-progress 7 Wonders session). Use `XCUIApplication.launchFresh(...)`.
- Drive elements by `accessibilityIdentifier` (e.g. `game.<id>`, `roster.<name>`,
  `setup.start`, `<category>.plus/.minus/.value`, `scoring.total`,
  `scoring.finish`, `results.edit/playAgain/done`, `home.resume`, `home.version`).

### SwiftData gotchas the tests guard against

- **`didSet`/`willSet` do NOT fire on `@Model` stored properties.** Bump
  `updatedAt` with explicit mutation methods (`rename(to:)`, `record(...)`,
  `softDelete()`, `complete(...)`) or a computed property whose setter calls
  `touch()` — never rely on a property observer.
- **In-memory stores can't use `@Attribute(.unique)`.** Tests that need a
  container use a fresh on-disk store in a temp directory, not
  `isStoredInMemoryOnly: true`.

## Layout

- `app/GameScoring/Models/` — SwiftData `@Model` entities
- `app/GameScoring/Games/` — `ScoringGame` protocol, game conformers, `SessionService`
- `app/GameScoring/Views/` — SwiftUI screens
- `app/GameScoring/Design/` — `Theme` tokens, shared components
- `app/GameScoringTests/` — Swift Testing suites
- `docs/` — PRD, UX design, style guide
